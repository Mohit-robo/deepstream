🤝 Phase 8 & 10 — apps/deepstream_nvtracker_v3.py

V3: Hybrid Tracking

Phase 8/10 combines DeepStream's native multi-object tracker (nvtracker / NvDCF) with the high-precision SUTrack TensorRT engine. It introduces a live OSD selection, Re-ID persistence, and a LOCKED/SEARCHING/STALE state machine.

Overview: Why Two Trackers?

TrackerStrengthWeakness
nvtracker (NvDCF) Multi-object, GPU-native, DeepStream-integrated, maintains global IDs IDs can drift during long occlusions; bounding boxes less precise than SOT
SUTrack (SOT) High-precision single-object tracking, robust to appearance change Single object only; no global ID; can drift if object leaves frame entirely
Hybrid (Phase 8) Precise SUTrack box + stable nvtracker ID correlation via IoU matching Requires PGIE for initial detection; more complex pipeline

Pipeline Architecture

GSTREAMER PIPELINE (deepstream_nvtracker_app.py) filesrc / v4l2src │ decodebin (nvv4l2decoder on Jetson) │ nvvideoconvert compute-hw=1 + capsfilter (NVMM RGBA) │ nvstreammux batch-size=1 │ nvinfer (PGIE — ResNet-10 INT8 detector) [interval set to 10000 after selection] │ nvtracker ll-lib=libnvds_nvdcf.so ll-config=nvdcf_config.txt │ → assigns native object_id to all detected objects each frame │ nvvideoconvert compute-hw=1 (bridge: nvinfer → nvdsosd format) │ nvdsosd │ ← PAD PROBE: read frame + nvtracker metadata → run SUTrack → write NvDsObjectMeta │ queue ── nv3dsink / fakesink [display branch]

What the Probe Does Each Frame

  1. Extract RGBA frame from NVMM buffer as NumPy array
  2. Convert RGBA → RGB for SUTrack preprocessing
  3. Run TrackerManager.update(frame_rgb, frame_idx) → SUTrack bounding boxes
  4. For each SUTrack box: iterate all NvDsObjectMeta (nvtracker objects), compute IoU
  5. If IoU > 0.5 with a native object → log the native ID correlation
  6. Write a new NvDsObjectMeta for the SUTrack result (teal box, "SUTrack [ID] (Stabilized)" label)

ID Stabilization Logic

STABILIZATION FLOW (per SUTrack output box) SUTrack output: [sx, sy, sw, sh] │ ▼ For each NvDsObjectMeta from nvtracker: nv_bbox = [rect.left, rect.top, rect.width, rect.height] iou = get_iou(sutrack_bbox, nv_bbox) if iou > max_iou: max_iou = iou best_match_meta = obj_meta │ ▼ if best_match_meta: log "ID Correction: SUTrack X matching native Y" # SUTrack's precise box "wins", native ID logged for correlation │ nvds_acquire_obj_meta → write SUTrack box + "Stabilized" label

The IoU threshold is 0.5 (hardcoded minimum to prevent false matches). The SUTrack box is always drawn regardless of whether a native match is found — the native ID correlation is informational only and does not block rendering.

Mid-Stream Target Switching

At any point during tracking, type s + ENTER in the terminal. An input_listener daemon thread sets state.trigger_reselect = True. The GStreamer probe detects this flag, captures the current frame + object metadata, and sets state.frame_ready to wake the select_loop thread.

MID-STREAM RESELECTION FLOW Terminal: user types "s" + ENTER │ input_listener thread: state.trigger_reselect = True │ GStreamer probe (next frame): captures frame_rgba + current detections state.frame_ready.set() │ select_loop thread (wakes on frame_ready): state.frame_ready.clear() select_bbox_click_to_select(frame, detections) → user clicks new target state.manager.active_trackers.clear() ← drop old tracker state.manager.initialize(frame_rgb, new_bbox, frame_idx=current_idx) │ Probe continues — now tracking new target
Critical: active_trackers must be cleared before re-init
Without state.manager.active_trackers.clear(), the old tracker continues running alongside the new one. Both compute SUTrack boxes every frame, doubling inference load and generating two overlapping "Stabilized" labels.
frame_ready must be cleared in main() after initial wait
After the first frame is captured and the user makes their initial selection, state.frame_ready.clear() MUST be called before the select_loop thread starts. Without this clear, select_loop wakes immediately (the event is still set from the initial probe) and shows the GUI again as if a reselection was requested.

Phase 10: deepstream_nvtracker_v3.py — Evolution

Phase 10 addresses three limitations of Phase 8: the pipeline-freeze during selection, permanent loss when a target re-enters the frame with a new ID, and the lack of clean auto-stop behaviour.

CapabilityPhase 8Phase 10 (V3)
Selection UI OpenCV click window; pipeline pauses Snap & Re-Sync; stdin controls
Target lost Drifts indefinitely SEARCHING → Re-ID → re-attach
Re-entry after frame exit Permanently lost IoU + histogram Re-ID
OSD feedback SUTrack [ID] (Stabilized) [LOCKED] / [SEARCHING...] / [TARGET LOST]

State Machine

PHASE 10 STATE MACHINE (deepstream_nvtracker_v3.py) IDLE ──(user selects)──► LOCKED ──(conf < 0.20 × 150f)──► SEARCHING ▲ │ │ (IoU >0.35 or hist >0.58) │ └─────────────── Re-ID ───────────────┘ │ (timeout 150f) ──► STALE │ ('s' keypress) ──► IDLE

Snap & Re-Sync: How Non-Blocking Selection Works

NON-BLOCKING SELECTION FLOW Timer (1 s after start) or 's' keypress: state.selection_active = True GStreamer probe (every frame while selection_active): → copies RGBA frame + detections to state.snapshot_frame/snapshot_dets → state.snapshot_ready.set() ← wakes GUI thread → returns immediately ← pipeline keeps running selection_thread: → shows click-to-select window on latest snapshot → user picks object → state.pending_init = {x, y, w, h, tracker_id} → state.selection_active = False GStreamer probe (next frame): → reads pending_init → looks up tracker_id in LIVE nvtracker metadata → initializes SUTrack from LIVE position (not stale snapshot) → state → LOCKED
Re-Sync advantage on live feeds
If the user takes 5 seconds to click, the nvtracker has been tracking the object for those 5 seconds. The SUTrack template is initialized from the current position, not from a 5-second-old snapshot. No video freeze, no coordinate jump.

Re-ID Strategy

Two-tier matching in _reid_scan():

  1. IoU spatial match (threshold 0.35) — object returned to roughly the same area. Fast, no pixel computation.
  2. Histogram appearance match (threshold 0.58) — H-S histogram (50×60 bins) of the candidate crop vs stored template. Uses Bhattacharyya distance converted to similarity.

[VisualTracker]
SearchRegionScale=1.6          # how wide to search for the object
FilterStrength=0.075           # lower = smoother but slower to adapt
MinTrackingConfidence=0.1      # below this threshold object is considered lost
