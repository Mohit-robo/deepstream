GStreamer Pipeline

How video flows from a file or camera through DeepStream. Covers both the current Phase 6 NvDsOsd+RTSP pipeline and the legacy Phase 4 appsink pipeline.

Phase 6 Pipeline — NvDsOsd + RTSP + PGIE (Current)

The recommended application is deepstream_rtsp_app.py. It uses a native DeepStream pad probe instead of appsink, draws boxes via nvdsosd in GPU memory, and streams output over RTSP using hardware H.264 encoding.

PHASE 6 PIPELINE (deepstream_rtsp_app.py) filesrc / v4l2src / uridecodebin | decodebin (hardware nvv4l2decoder on Jetson) | nvvideoconvert compute-hw=1 VIC: NVMM → NVMM RGBA (no EGL context) | nvstreammux batch-size=1 width=1280 height=720 | nvinfer config-file-path=pgie_config.txt [Phase 6: optional PGIE detector] | ResNet-10 INT8; writes NvDsObjectMeta detections into frame metadata | nvvideoconvert compute-hw=1 bridge required between nvinfer → nvdsosd | nvdsosd process-mode=0 | | ← PAD PROBE fires on sink pad (Python tracker_probe callback): | - frame 0: capture RGBA frame + read PGIE detections from obj_meta_list | state.frame_ready.set() signal main thread | state.init_done.wait() block until ROI is selected | - frame 1+: cv.cvtColor RGBA→RGB | TrackerManager.update() | write NvDsObjectMeta bbox + label per tracked object | +──── queue ──── nvvideoconvert(hw=1) ──── nv3dsink [local display] | └──── queue ──── nvvideoconvert(hw=1) ──── nvv4l2h264enc | rtph264pay | udpsink port=5400 GstRtspServer port=8554: udpsrc(5400) → rtsp://<jetson-ip>:8554/sutrack

Pad Probe Logic — tracker_probe()

The pad probe is attached to the sink pad of nvdsosd. It fires after nvinfer has already populated frame metadata, but before nvdsosd renders the frame. This ordering ensures PGIE detections are available when the probe reads them on frame 0.

1

First Frame — Capture & Signal

On frame 0, the probe calls pyds.get_nvds_buf_surface() to get an RGBA NumPy array, reads all NvDsObjectMeta entries from obj_meta_list (populated by nvinfer), stores them in state.first_frame_detections, and calls state.frame_ready.set() to wake the main thread. Then it blocks on state.init_done.wait() until ROI selection is complete.

2

Main Thread — ROI Selection (while probe waits)

The main thread waits on state.frame_ready.wait(), then calls select_bbox_click_to_select() or manual_roi_select() (see PGIE page). All OpenCV GUI calls happen here — never inside the probe (Lesson 25). Once a box is selected, state.init_done.set() unblocks the probe.

3

Subsequent Frames — Track & Draw

The probe converts the RGBA frame to RGB, calls TrackerManager.update(), and for each returned bounding box acquires an NvDsObjectMeta from the pool, fills rect_params and text_params, and adds it to the frame. nvdsosd then renders these boxes in GPU memory before the tee.

Lesson 25 — OpenCV GUI in Main Thread Only
cv2.namedWindow, cv2.imshow, cv2.setMouseCallback, and cv2.selectROI must never be called inside a GStreamer pad probe. The probe runs in a GStreamer streaming thread; Qt requires GUI calls on the main thread. Use threading.Event to synchronize (Lesson 26).

RTSP Streaming

After nvdsosd, a tee element splits the stream into two branches. The RTSP branch encodes with nvv4l2h264enc (Jetson NVENC hardware block) and sends RTP packets to a udpsink on port 5400. A GstRtspServer instance wraps a udpsrc media factory and serves the combined stream at rtsp://<jetson-ip>:8554/sutrack.

The display branch tries nv3dsink (Jetson-native, no EGL copy) first, falling back to nveglglessink with a nvvideoconvert(compute-hw=1) bridge to avoid the NVMM surface array crash (Lesson 24).

Headless / SSH Mode
Set pipeline.headless: true in tracker_config.yml. The display branch connects to fakesink instead of nv3dsink. No DISPLAY environment variable is needed. All output goes to the RTSP stream. Combine with tracker.static_roi for fully unattended operation.

Phase 4 (Legacy) — What is GStreamer?

GStreamer is a media framework that chains together elements like LEGO bricks to form a pipeline. Each element does one job: reading a file, decoding video, converting colors, or handing frames to Python.

On Jetson, NVIDIA provides special elements prefixed with nv that use the hardware video decoder and GPU for color conversion — far faster than doing it in software.

Why DeepStream?
DeepStream is NVIDIA's SDK built on GStreamer. It gives hardware-accelerated decode (nvv4l2decoder) and color conversion (nvvideoconvert) for free, leaving the CPU entirely available for the tracking logic.

Building the Pipeline

The entire pipeline is written as a single string and parsed by Gst.parse_launch(). Elements are connected with ! — just like Unix pipes.

PIPELINE STRING (file input) filesrc location=/path/to/video.mp4 ! decodebin # auto-selects nvv4l2decoder on Jetson (HW decode) ! nvvideoconvert compute-hw=0 # GPU: NVMM memory -> system RAM, output RGBA ! video/x-raw,format=RGBA # capability filter: force RGBA ! videoconvert # CPU: final format normalization ! video/x-raw,format=RGBA ! appsink name=appsink # hand buffer to Python callback emit-signals=true # fire on_new_sample() each frame max-buffers=1 # keep only the latest frame drop=true # drop old frames if tracker is slow sync=false # process as fast as possible

Why Two Color Converters?

The Jetson hardware decoder outputs frames in NVMM memory — GPU memory the CPU cannot read directly. Two elements are needed to bridge this:

1

nvvideoconvert — NVMM to System RAM

NVIDIA element that moves the frame from GPU memory to CPU-accessible RAM and converts to RGBA. compute-hw=0 uses the GPU path, which requires an EGL context (DISPLAY=:0). Use compute-hw=1 (VIC) for SSH headless sessions where no display is connected.

2

videoconvert — Format normalization

Standard CPU-based GStreamer converter. Ensures the frame at appsink is always RGBA regardless of what the upstream source produces.

Buffer Conversion — nvbuf_to_bgr()

Each GStreamer frame arrives as a raw GstBuffer. This function maps it into a NumPy array and converts to BGR (OpenCV's native format). 

def nvbuf_to_bgr(sample) -> np.ndarray:
    buf       = sample.get_buffer()
    structure = sample.get_caps().get_structure(0)

    width  = structure.get_int('width')[1]
    height = structure.get_int('height')[1]
    fmt    = structure.get_string('format')   # e.g. "RGBA"

    result, mapinfo = buf.map(Gst.MapFlags.READ)
    data = np.frombuffer(mapinfo.data, dtype=np.uint8)
    buf.unmap(mapinfo)

    # Most common path on Jetson after nvvideoconvert
    if fmt in ('RGBA', 'RGBx'):
        img = data.reshape(height, width, 4)
        return cv.cvtColor(img, cv.COLOR_RGBA2BGR)

Supported pixel formats:

FormatChannelsConversion
RGBA, RGBx4 (alpha unused)RGBA → BGR via OpenCV
RGB3RGB → BGR via OpenCV
BGRx, BGRA4 (alpha unused)Slice first 3 channels
BGR3Copy as-is
NV12, I420YUV planarYUV → BGR via OpenCV
Why return BGR and not RGB?
OpenCV works natively in BGR. We return BGR here, then convert to RGB right before calling the tracker. Keeping these steps separate makes intent clear: buffer decoding returns OpenCV-native BGR; the tracker always receives RGB (training format).

Application State — AppState

AppState holds everything the callback needs. Because GStreamer fires callbacks from its own context, using a class instance avoids global variables. 

class AppState:
    manager:      TrackerManager   # runs SUTrack inference
    cfg:          dict              # loaded YAML config
    output_path:  str               # .mp4 save path, or empty string
    headless:     bool              # True = no display window

    initialized:  bool              # has ROI been selected yet?
    frame_idx:    int               # frames sent to tracker (stale removal)
    frame_count:  int               # frames with results (FPS averaging)
    total_time:   float             # cumulative tracker wall-clock time

    writer:       cv.VideoWriter    # MP4 writer — created lazily on first frame
    loop:         GLib.MainLoop     # so callback can quit the loop on 'q'
Lazy VideoWriter
The VideoWriter is created on the first frame because the video resolution is only known after the pipeline starts negotiating formats with the decoder.

Appsink Callback — on_new_sample()

GStreamer calls this function once for every decoded frame.

on_new_sample() FLOW GStreamer delivers frame | v nvbuf_to_bgr() --> NumPy BGR array | +-- first frame? --YES --> imshow() + selectROI() (GUI mode) | OR --init_bbox value (headless) | --> manager.initialize(rgb, bbox) | return OK | v (subsequent frames) cvtColor BGR->RGB Critical Rule 5: model was trained on RGB | manager.update() --> { obj_id: [x, y, w, h] } | draw green rectangle + "ID N" for each tracked object draw red FPS counter top-left: "24.8 FPS | tracks: 1" | +-- VideoWriter.write() (if --output was set) +-- cv.imshow() (if not --headless) 'q' pressed? --> loop.quit() | return Gst.FlowReturn.OK

First Frame — Initializing the Tracker

The tracker needs a template before it can track anything. Two ways to provide the initial bounding box:

A

GUI Mode (default)

cv.imshow() is called before cv.selectROI() to initialize the Qt window handler. Skipping this causes a NULL window crash on some OpenCV builds. Draw a rectangle and press SPACE or ENTER.

B

Headless Mode (--init_bbox X Y W H)

Skips GUI entirely. Box comes from the command line. Ideal for CI, automated benchmarks, or SSH sessions without a display.

Drawing Results

After update() returns a dict of { id → [x, y, w, h] }, each box is drawn as a green rectangle with the object ID above it. A red FPS counter in the top-left shows tracking speed and active track count.

Wraps the engine with multi-object lifecycle logic. Passes crop factors, thresholds, max_age, Hanning flag.

Sets pipeline to PLAYING and blocks in loop.run() until EOS, error, or q. Then cleans up pipeline, VideoWriter, and OpenCV windows.

SSH Headless Tip
No monitor connected? Use nvvideoconvert compute-hw=1 (VIC mode, no EGL required) to avoid the "nvbufsurface: Failed to create EGLImage" error. Or set export DISPLAY=:0 if a physical monitor is attached to the Jetson.
