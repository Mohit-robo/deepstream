Phase 11 (V4) wraps the tracking pipeline inside a native Linux GTK3 application. It introduces GUI control buttons and unlocks full hardware FPS during tracking via a decoupled asynchronous TensorRT worker thread.

FPS Optimization: TRTWorkerThread

Root Cause: Probe Blocking the GStreamer Streaming Thread

In Phase 10, manager.update() (SUTrack TRT inference, ~30 ms on Jetson) ran directly inside the GStreamer pad probe. A pad probe runs on the streaming thread — any blocking work inside it stalls the entire pipeline until the probe returns.

Solution: TRTWorkerThread

All manager mutations (initialize, update, clear) run exclusively in TRTWorkerThread. The probe submits frames non-blocking via a pending-slot double-buffer and reads the last result for OSD.

ASYNC TRT WORKER FLOW GStreamer probe (every frame): TRTWorkerThread (daemon): get_rgb() (~3 ms) wait for trigger submit_track(rgb) (<0.1 ms) -----> preprocess + TRT infer (~30 ms) read last_result (GIL-safe) store last_result write OSD box (~1 ms) <----- (probe reads on next call) return OK (total ~4 ms) If worker is still busy when the next frame arrives, the pending slot is overwritten -- always the freshest frame wins.
1-2 frame OSD lag is imperceptible
The displayed bbox reflects TRT results from 1-2 frames ago. At 60 FPS this is 16-33 ms of positional lag. For a human walking at normal speed, the bbox moves < 2 pixels in that time. The visual difference is undetectable.

Thread Safety

  • _lock protects _pending (written by probe, read by worker)
  • last_result / last_conf replaced atomically — Python dict assignment is GIL-safe; probe reads without holding any lock
  • All manager calls go through the worker — TrackerInstance.state is never written from two threads simultaneously
  • Button handlers call trt_worker.submit_init() / submit_clear() — never touch manager directly
Lesson 34: Never run TRT inference inside a GStreamer pad probe
A pad probe runs on the GStreamer streaming thread. Any blocking operation (> 2 ms) inside the probe caps pipeline throughput to 1 / block_time FPS. Use a worker thread with a pending-slot double-buffer: submit the frame, return immediately, read the last result for OSD.
