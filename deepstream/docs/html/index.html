SUTrack DeepStream Tracker

A production-ready, real-time object tracking application built on NVIDIA DeepStream and GStreamer. Runs on Jetson hardware with zero PyTorch dependency at runtime — pure NumPy preprocessing and a TensorRT FP32 engine.

What Does This App Do?

SUTrack is an AI model that can track an object across a video, even when the object moves, changes scale, or gets partially hidden. You point it at an object in the first frame — either by clicking on a detector's suggestion (Phase 6) or by drawing a box — and it follows that object through every subsequent frame.

This DeepStream application wraps SUTrack in a hardware-accelerated video pipeline so it runs efficiently on NVIDIA Jetson devices. Here is the big picture:

END-TO-END FLOW ┌─────────────────────────────────────────────────────────────────────────┐ │ VIDEO SOURCE → HW DECODE → [PGIE DETECT] → TRACKER → OUTPUT │ │ file / camera Jetson GPU ResNet-10 INT8 SUTrack RTSP stream │ RTSP stream nvv4l2decoder click-to-select TRT engine nv3dsink └─────────────────────────────────────────────────────────────────────────┘ Steps at startup: 1. Pipeline starts; PGIE (ResNet-10 detector) runs on the first decoded frame 2. A window shows the frame with all detected objects boxed — user clicks one 3. The clicked box is passed to the SUTrack engine as the initial template Steps per frame (after init): 1. GStreamer pad probe fires with each decoded frame in GPU (NVMM) memory 2. Frame is read as an RGBA NumPy array and converted to RGB 3. SUTrack TRT engine predicts a new bounding box for the tracked object 4. Box metadata is written to NvDsObjectMeta → nvdsosd renders it in GPU memory 5. Output tee: local display (nv3dsink) + RTSP stream (H.264 NVENC)

Full Pipeline Diagram (Phase 6)

GSTREAMER PIPELINE (deepstream_rtsp_app.py) filesrc location=video.mp4 │ decodebin (auto-selects nvv4l2decoder on Jetson — hardware decode) │ nvvideoconvert compute-hw=1 (VIC: NVMM → NVMM RGBA, no EGL context needed) │ nvstreammux batch-size=1 width=1280 height=720 │ nvinfer config-file-path=pgie_config.txt [optional Phase 6 PGIE detector] │ nvvideoconvert compute-hw=1 (bridge: required between nvinfer and nvdsosd) │ nvdsosd process-mode=0 │ ← PAD PROBE fires here (Python): read frame, update tracker, write NvDsObjectMeta │ ├──── queue ──── nvvideoconvert(hw=1) ──── nv3dsink / nveglglessink [display] │ └──── queue ──── nvvideoconvert(hw=1) ──── nvv4l2h264enc ──── rtph264pay ──── udpsink GstRtspServer: udpsrc → rtsp://<jetson-ip>:8554/sutrack

Quick Start

Step 1 — Build the TensorRT Engine (one time only)

# From the SUTrack repo root
cd ~/SUTrack

# Export the model to ONNX format
python tracking/export_onnx.py --param sutrack_t224

# Compile the TRT engine (FP32 — do NOT add --fp16)
bash deepstream/scripts/build_engine.sh sutrack.onnx sutrack_fp32.engine
Critical: FP32 Only
Never compile with --fp16. The sigmoid activation inside the TRT graph produces values close to 0 in FP16 precision, making the tracker see no peaks and output a broken bounding box.

Step 2 — Run with PGIE Click-to-Select ROI (Phase 6, recommended)

export DISPLAY=:0   # required for OpenCV window
python deepstream/apps/deepstream_rtsp_app.py \
    --config deepstream/configs/tracker_config.yml \
    --input /path/to/video.mp4

A window titled "Click-to-Select Object" appears on the first frame with all detector boxes drawn. Click on the object you want to track. The RTSP stream starts at rtsp://<jetson-ip>:8554/sutrack.

Step 3 — Run headless with static ROI (no display needed)

# Set in tracker_config.yml:
#   tracker.static_roi: "300,200,120,140"
python deepstream/apps/deepstream_rtsp_app.py \
    --config deepstream/configs/tracker_config.yml \
    --input /path/to/video.mp4

Critical Rules

These rules come from hard-won debugging experience. Breaking any one of them produces subtle, hard-to-diagnose bugs.

Rule 1 — FP32 Engine Only
Always compile the TRT engine without --fp16. FP16 underflows sigmoid outputs to near-zero → tracker breaks.
Rule 2 — No Double Sigmoid
score_map and size_map are already sigmoid-activated inside the TRT graph. Applying sigmoid again in Python produces values stuck at ~0.5 — completely wrong.
Rule 3 — Position-Only Clamp
When clamping the output bounding box to image boundaries, clamp only x1, y1. Never re-derive w, h from clamped corners — that causes the box to shrink to a line when the target is near the edge.
Rule 4 — Hanning Window Before argmax
Apply the Hanning spatial penalty to the score map before finding the peak with argmax. Doing it after has no effect.
Rule 5 — BGR → RGB Before Every Tracker Call
GStreamer and OpenCV produce BGR images. The model was trained on RGB. Convert with cv.cvtColor(frame, cv.COLOR_BGR2RGB) before passing any frame to the tracker.
Rule 6 — Named TRT Bindings
Always access TensorRT tensors by name (inputs['template']), never by numeric index. Index order can change between engine versions.
