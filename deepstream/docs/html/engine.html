TensorRT Engine

How the SUTrack AI model is loaded onto the GPU and run efficiently using TensorRT. This is where frames become bounding box predictions.

What is TensorRT?

TensorRT is NVIDIA's inference optimizer. It takes a trained neural network (exported as ONNX) and compiles it into a highly optimized engine file tuned for a specific GPU. The engine runs much faster than a plain PyTorch model because TensorRT fuses layers, picks optimal kernel implementations, and pre-allocates memory.

Once the engine is compiled (a one-time step), inference only needs: TensorRT, cuda-python (cuda.cudart), and NumPy — no PyTorch required.

ENGINE LIFECYCLE Training (PyTorch) | v export_onnx.py sutrack.onnx | v trtexec (one-time compile, ~3 min on Jetson Orin) sutrack_fp32.engine <-- lives at ~/SUTrack/ (repo root) | v SUTrackEngine.__init__() (at app startup) deserialized TRT runtime + execution context | v SUTrackEngine.infer() (called every frame, per tracker) score_map, size_map, offset_map, pred_boxes

Buffer Allocation — HostDeviceMem and _allocate_buffers()

The Memory Problem

TensorRT runs on the GPU. To feed it data, we need to:

  1. Allocate pinned host memory — CPU RAM that the GPU can DMA from directly (faster than regular RAM)
  2. Allocate device memory — memory on the GPU itself
  3. Copy data host → device before inference, device → host after

HostDeviceMem — Paired Buffer

A simple container that keeps the host and device pointers for one tensor together: 

class HostDeviceMem:
    # One tensor's paired CPU + GPU memory
    host:   np.ndarray   # pinned NumPy array (CPU side)
    device: int          # raw GPU pointer (device side)
    name:   str          # tensor name e.g. 'template', 'score_map'

_allocate_buffers() — How Memory Is Allocated

This function is called once at engine startup. It loops over every tensor in the engine (inputs and outputs) and allocates both a pinned host buffer and a GPU device buffer. 

for i in range(engine.num_io_tensors):
    name  = engine.get_tensor_name(i)        # by name, never by index
    dtype = trt.nptype(engine.get_tensor_dtype(name))
    shape = engine.get_tensor_shape(name)

    # Allocate pinned host memory (CPU, DMA-accessible)
    err, host_ptr = cudart.cudaMallocHost(nbytes)
    host_np = np.frombuffer(
        (ctypes.c_char * nbytes).from_address(int(host_ptr)),
        dtype=dtype).reshape(shape)

    # Allocate device memory (GPU)
    err, device_ptr = cudart.cudaMalloc(nbytes)

    # Sort into inputs or outputs dict
    is_input = engine.get_tensor_mode(name) == trt.TensorIOMode.INPUT
    (inputs if is_input else outputs)[name] = HostDeviceMem(host_np, device_ptr, name)
Why cuda.cudart instead of pycuda?
pycuda requires a compiled extension and has compatibility issues on some Jetson JetPack versions. cuda-python (from cuda import cudart) is a thin Python wrapper around the CUDA runtime C library — it is included in JetPack and works reliably on all Jetson devices.

The three input tensors and their shapes:

NameShapeWhat it contains
template(1, 6, 112, 112)Normalized template crop of the target object (float32)
search(1, 6, 224, 224)Normalized search region around the previous position (float32)
template_anno(1, 4)Normalized [x1, y1, w, h] of the target inside the template crop (float32)

The four output tensors:

NameShapeWhat it contains
score_map(1, 1, 14, 14)Confidence score for each location in the 14x14 feature grid. Already sigmoid-activated inside the graph.
size_map(1, 2, 14, 14)Predicted normalized width and height at each grid cell.
offset_map(1, 2, 14, 14)Sub-pixel x/y offset from each grid cell center.
pred_boxes(1, 4)Direct box prediction (used as a cross-check; postprocessing uses the maps above).
Critical Rule 2 — No Double Sigmoid
score_map and size_map are already sigmoid-activated inside the TRT computation graph. Do not call sigmoid() on them in Python. Doing so maps already-small values (e.g. 0.6) through sigmoid again, producing values stuck near 0.5 — the tracker sees a flat score map and tracks nothing correctly.

SUTrackEngine Class

SUTrackEngine is created once at startup and shared by all TrackerInstance objects. There is only one TRT context and one set of buffers. Inference calls are made sequentially — one tracker at a time per frame. 

class SUTrackEngine:
    def __init__(self, engine_path, template_size=112,
                 search_size=224, encoder_stride=16):

        self.feat_sz = search_size // encoder_stride  # 224 // 16 = 14

        # Load and deserialize engine
        with open(engine_path, 'rb') as f, trt.Runtime(TRT_LOGGER) as runtime:
            self.engine = runtime.deserialize_cuda_engine(f.read())

        # Create execution context and set fixed input shapes
        self.context = self.engine.create_execution_context()
        self.context.set_input_shape('template',      (1, 6, 112, 112))
        self.context.set_input_shape('search',        (1, 6, 224, 224))
        self.context.set_input_shape('template_anno', (1, 4))

        # Allocate all GPU buffers once
        self.inputs, self.outputs, self.bindings, self.stream = \
            _allocate_buffers(self.engine, template_size, search_size)
Shared Engine Design
All TrackerInstance objects share this one engine. Each tracker copies its own template and search crops into the shared buffers before calling infer(). This is correct because the engine is stateless — the per-object state (template image, bounding box) lives in Python, not inside TensorRT. The benefit is that we avoid duplicating the large engine weights in GPU memory for each tracked object.

The infer() Method

This is the method every TrackerInstance calls once per frame. It does four things:

1

Copy inputs to host buffers

Uses np.copyto() to fill the pre-allocated pinned host arrays with the current tracker's template, search crop, and annotation.

2

Host to Device (H2D) copy

Asynchronously copies each input from pinned CPU RAM to GPU memory using cudaMemcpyAsync on the CUDA stream.

3

TRT forward pass

context.execute_async_v2() runs the neural network on the GPU asynchronously on the same CUDA stream.

4

Device to Host (D2H) copy + sync

Copies each output tensor back to host memory, then cudaStreamSynchronize() waits for all GPU operations to complete before returning.

def infer(self, template_img, search_img, template_anno) -> dict:
    # Step 1: copy inputs into shared host buffers
    np.copyto(self.inputs['template'].host,      template_img)
    np.copyto(self.inputs['search'].host,        search_img)
    np.copyto(self.inputs['template_anno'].host, template_anno)

    # Step 2: H2D — host to device (async)
    for inp in self.inputs.values():
        cudart.cudaMemcpyAsync(inp.device, inp.host.ctypes.data,
            inp.host.nbytes, cudaMemcpyHostToDevice, self.stream)

    # Step 3: TRT forward pass (async)
    self.context.execute_async_v2(bindings=self.bindings,
                                    stream_handle=self.stream)

    # Step 4: D2H — device to host (async) + sync
    for out in self.outputs.values():
        cudart.cudaMemcpyAsync(out.host.ctypes.data, out.device,
            out.host.nbytes, cudaMemcpyDeviceToHost, self.stream)
    cudart.cudaStreamSynchronize(self.stream)

    return {name: mem.host for name, mem in self.outputs.items()}

Why FP32 Only?

The SUTrack decoder uses sigmoid activations to produce confidence scores and size predictions. In float32, sigmoid outputs a value between 0 and 1 as expected. In float16, values close to zero underflow to exactly 0 due to limited precision.

SIGMOID UNDERFLOW IN FP16 Example: logit value = -3.2 FP32 sigmoid: 1 / (1 + exp(3.2)) = 0.0391 correct small confidence FP16 sigmoid: 1 / (1 + exp(3.2)) = 0.0000 underflows to zero Effect: FP32 score_map: values in [0.01 .. 0.85] --> clear peak visible FP16 score_map: all values = 0.000 --> no peak, tracker broken
Critical Rule 1 — FP32 Engine Only
Always compile the engine without --fp16. The correct trtexec command is: 
trtexec --onnx=sutrack.onnx --saveEngine=sutrack_fp32.engine \
    --minShapes=template:1x6x112x112,search:1x6x224x224,template_anno:1x4 \
    --optShapes=template:1x6x112x112,search:1x6x224x224,template_anno:1x4 \
    --maxShapes=template:1x6x112x112,search:1x6x224x224,template_anno:1x4 \
    --memPoolSize=workspace:4096MiB

Why Named TRT Bindings?

Older TensorRT APIs accessed tensors by numeric index (0, 1, 2 ...). The index order could change between TRT versions or engine recompilations, silently feeding data to the wrong tensor.

This engine always uses named access: 

# WRONG — index order is fragile
self.inputs[0].host = template_img

# CORRECT — name is stable across TRT versions
self.inputs['template'].host = template_img
Critical Rule 6 — Named Bindings Always
Access TRT tensors by name, never by numeric index. Index order is an implementation detail that can change silently.
