🌐 Phase 12 — apps/deepstream_server_app.py + v5_remote_client.py

V5: Remote Client-Server

Phase 12 (V5) separates compute from the operator. The Jetson runs a fully headless server — no monitor, no display server, no X11 required — streaming annotated video over RTSP while exposing a REST control API. The operator connects from any PC using a lightweight GTK client, sees the live stream, and clicks to select tracking targets.

Architecture

V5 SYSTEM OVERVIEW JETSON (headless SSH) OPERATOR PC ================================ ============================== deepstream_server_app.py v5_remote_client.py | | +--> filesrc / v4l2src +--> rtspsrc (TCP, port 8554) +--> decodebin +--> rtph264depay +--> nvvideoconvert +--> h264parse + avdec_h264 +--> nvstreammux +--> videoconvert +--> nvinfer (PGIE ResNet-10) ----------+ +--> GTK DrawingArea (Cairo) +--> nvtracker (NvDCF) | | +--> nvdsosd (OSD annotations) <--------+ +--> REST client thread +--> tee | GET /api/state (0.5 s poll) |--> fakesink (headless branch) | POST /api/command |--> nvv4l2h264enc | --> rtph264pay --> udpsink click --> POST /api/command {click, x, y} | key --> POST /api/command {select/lock/...} GstRtspServer (port 8554) <----+ REST API (port 8000) <-------- <----- HTTP

The two processes communicate over two independent channels:

  • RTSP stream (port 8554, TCP): annotated H.264 video — OSD boxes, state banners, confidence scores — rendered entirely on the Jetson GPU.
  • REST API (port 8000, HTTP): low-rate control channel. Client polls state every 0.5 s and POSTs commands (click, select, lock, cancel). Note: The /api/state endpoint also continuously broadcasts the raw Bounding Box coordinates to the client for programmatic extraction.

Headless EGL Fix: EGL_PLATFORM=surfaceless

nvinfer (PGIE) on Jetson uses EGL to create EGLImage objects from NVMM (Surface Array) buffers during inference. This requires a usable EGL context — which in turn requires a GPU device connection. In a normal SSH session without X11, no display is available and the pipeline fails immediately:

FAILURE MODE (unpatched) $ ssh phoenix@jetson $ python3 deepstream_server_app.py ... nvbufsurface: Failed to create EGLImage. WARN nvinfer: Failed to set buffer pool to active ERROR: Pipeline failed to enter PLAYING state.

Why X11 Forwarding Does Not Help

SSH with -X sets DISPLAY=localhost:10.0 — a forwarded X11 socket tunnelled back to the operator PC. NVIDIA's libEGL on Jetson cannot use this because:

  • EGLImage creation for NVMM buffers requires a local DRM/tegra device.
  • The forwarded X11 socket has no backing local GPU device — it is a network connection to a remote X server.
  • xdpyinfo reports the display as reachable (X11 protocol works), but the EGL layer never sees a local framebuffer.
Do not test EGL usability with xdpyinfo
xdpyinfo tests X11 protocol connectivity, not GPU device access. A display that passes xdpyinfo can still fail EGLImage creation if it is a forwarded socket.

The Fix

The server drops any forwarded DISPLAY and sets EGL_PLATFORM=surfaceless before any GStreamer or DeepStream import. This uses the EGL_EXT_platform_device extension to bind EGL directly to the GPU device — no window system required at all.

STARTUP ENVIRONMENT SETUP (deepstream_server_app.py) # 1. Drop forwarded DISPLAY (e.g. 'localhost:10.0') disp = os.environ.get('DISPLAY', '') if disp and not disp.startswith(':'): # remote host prefix = forwarded del os.environ['DISPLAY'] # 2. Surfaceless EGL -- GPU device direct, no display server needed os.environ['EGL_PLATFORM'] = 'surfaceless' # 3. DeepStream headless hints os.environ['NV_DS_HEADLESS'] = '1' os.environ['QT_QPA_PLATFORM'] = 'offscreen' # (Must run before any GStreamer/DeepStream import and before LD_PRELOAD re-exec)
Lesson 36: EGL_PLATFORM=surfaceless for headless Jetson
For any headless Jetson DeepStream deployment (SSH, Docker, CI), always set EGL_PLATFORM=surfaceless and remove any forwarded DISPLAY. This is NVIDIA's recommended mechanism for display-free inference. It must be set before the first GStreamer import and must be inherited by any child process (set before LD_PRELOAD re-exec).

PGIE Auto-Detect: No ROI Drawing

In V4, the operator first sees a blank (or frozen) screen and must draw a bounding box to initialize the tracker. V5 eliminates this — the ResNet-10 PGIE detector runs every frame and automatically annotates all detected objects on the RTSP stream. The operator simply clicks on the object they want to track; no bounding box drawing is ever needed.

Detection-Driven IDLE State

When the server is in IDLE state (no target locked), the pad probe draws all current detections as light-blue labeled boxes and displays a status banner. The operator sees exactly what the detector sees in real-time:

IDLE STATE OSD (PGIE active, 3 objects detected) +-------------------------------------------------------+ | [3 DETECTED] Click on target in client | | | | +--[Person]--+ +--[Person]--+ +-[Person]-+ | | | | | | | | | | +------------+ +------------+ +----------+ | | | +-------------------------------------------------------+

Click-to-Select Flow

When the operator clicks anywhere on the client video canvas, the client sends normalized coordinates {x: 0.0-1.0, y: 0.0-1.0} to the server. The server:

  1. Denormalizes the click using its own live frame dimensions.
  2. Finds the detection whose centroid is nearest to the click point (Euclidean distance).
  3. Queues _pending_lock — processed on the next probe call with the live frame.
  4. Initializes SUTrack with that detection's bounding box and enters LOCKED state.
Normalized coordinates compensate for RTSP latency
The RTSP stream has ~0.5-1.5 s of buffering latency. If the client sent raw pixel coordinates, they would correspond to a frame that the server processed 1.5 s ago — the target has moved on. By sending normalized spatial cues ("near the left third, upper half") and resolving them to current live detections on the server side, the click selects the right object regardless of latency.

Auto-Lock Mode (--auto-lock)

For fully automated pipelines where no operator interaction is available, --auto-lock causes the server to pick the largest detected object (by bounding-box area) on the first frame that contains detections and initialize tracking immediately — zero human input.

Server Pipeline

SERVER PIPELINE (deepstream_server_app.py) filesrc/v4l2src | decodebin (dynamic pad-added) | nvvideoconvert [nvbuf-memory-type=4, NV12] | capsfilter [video/x-raw(memory:NVMM), NV12] | nvstreammux [batch=1, WxH from config] | nvinfer (PGIE) [ResNet-10 INT8 -- detections every frame] | nvtracker (NvDCF) [multi-object ID assignment] | +-----------pad probe (tracker_probe)----------+ | - _capture_dets() read NvDsObjectMeta | | - IDLE: draw all detections + banner | | - LOCKED: submit_track() to TRTWorkerThread | | - STALE: draw detections + re-acquire msg | +----------------------------------------------+ | nvvideoconvert [RGBA bridge] | capsfilter [video/x-raw(memory:NVMM), RGBA] | nvdsosd [process-mode=0 CPU] | tee +-- queue --> fakesink (headless: discards rendered frames) +-- queue --> nvvideoconvert --> nvv4l2h264enc [FP32, iframeinterval=30] --> rtph264pay [pt=96, config-interval=1] --> udpsink [127.0.0.1:5400] GstRtspServer factory: udpsrc:5400 ! rtph264depay ! rtph264pay name=pay0 pt=96 mount: rtsp://0.0.0.0:8554/sutrack

TRTWorkerThread

SUTrack TRT inference (~30 ms on Jetson) runs in a dedicated background thread — never in the GStreamer probe. The probe submits frames non-blocking via a pending-slot and reads the last computed result for OSD. This allows the pipeline to run at full source rate (~60 FPS) while the tracker runs at its hardware-limited rate (~30 FPS).

Client Pipeline

CLIENT PIPELINE (v5_remote_client.py) rtspsrc [location=rtsp://<host>:8554/sutrack, protocols=tcp, latency=200] | rtph264depay | h264parse | avdec_h264 (software decode on PC) | videoconvert | capsfilter [video/x-raw, format=RGB] | appsink [emit-signals=true, max-buffers=1, drop=true, sync=false] | on_new_sample() callback --> convert to numpy array --> scale to fit GTK DrawingArea (aspect-preserving letterbox) --> draw via Cairo surface --> GTK invalidate (GLib.idle_add)

TCP Transport

rtspsrc protocols=tcp forces RTP-over-TCP interleaving. All media data travels over the existing port 8554 TCP connection — no additional UDP ports are allocated. This is essential when any firewall or NAT exists between the Jetson and the operator PC.

Default rtspsrc uses UDP — will fail through firewalls
Without protocols=tcp, rtspsrc allocates random UDP ports for RTP media transport. These ports are almost always blocked by firewalls on corporate or home networks. Always use protocols=tcp (or protocols=0x4) for remote deployments.

Auto-Reconnect

If the RTSP connection is lost (server restart, network glitch), the client automatically retries after 5 seconds using GLib.timeout_add_seconds(5, self._reconnect). The pipeline is set to NULL, the frame buffer is cleared, and a new rtspsrc pipeline is constructed. The GTK window shows a black canvas until the connection is restored.

REST API

The server exposes a minimal HTTP API on port 8000.

GET /api/state

Returns the current tracking state. Polled by the client every 0.5 s to update the status bar.

FieldTypeDescription
statestringidle | selecting | locked | searching | stale

target_idintNvDCF tracker ID of locked target (-1 = none)
frame_idxintMonotonic frame counter since pipeline start
bboxlist[x, y, w, h] coordinates of tracked target (if locked/searching)

POST /api/command

Sends a control command. Body is JSON.

ActionExtra fieldsEffect

select—Enter SELECTING mode — OSD highlights first candidate
next—Cycle selection to next detection
prev—Cycle selection to previous detection
lock—Initialize SUTrack on currently highlighted detection
cancel—Stop tracking, return to IDLE

Running V5

Prerequisites

  • Jetson: DeepStream 6.x, TRT engine at ~/SUTrack/sutrack_fp32.engine
  • Jetson: Python packages: pyds, gi, numpy, opencv-python
  • Operator PC: python3-gi, gstreamer1.0-plugins-*, gstreamer1.0-libav, gir1.2-gtk-3.0, requests

Step 1 — Start the Server (Jetson)

cd ~/SUTrack python3 deepstream/apps/deepstream_server_app.py \ --config deepstream/configs/tracker_config.yml \ --input /path/to/video.mp4 \ --loop # Expected startup output: # [server] Dropping X11-forwarded DISPLAY=localhost:10.0 (not usable by nvinfer). # [server] EGL_PLATFORM=surfaceless (headless nvinfer -- no display required). # [INFO] tracker.sutrack_engine: SUTrackEngine ready. feat_sz=14 # [INFO] main: RTSP stream: rtsp://192.168.x.x:8554/sutrack # [INFO] main: REST API: http://192.168.x.x:8000/api/

Step 2 — Start the Client (Operator PC)

python deepstream/apps/v5_remote_client.py --host 192.168.x.x # The GTK window opens showing the RTSP stream. # Detection boxes appear in blue labelled with class names. # Click on the object you want to track. # Tracking begins immediately -- box turns green with confidence score. # EXTRACTING BOUNDING BOXES: # The client automatically prints the live tracked coordinates to the terminal: # BBOX,1452,320.5,150.2,85.0,210.0 # You can easily pipe this: # python deepstream/apps/v5_remote_client.py --host 192.168.x.x | grep "BBOX" > out.csv

Fully Automatic (no operator interaction)

python3 deepstream/apps/deepstream_server_app.py \ --config deepstream/configs/tracker_config.yml \ --input /path/to/video.mp4 \ --auto-lock --loop

With --auto-lock, the server locks onto the largest PGIE detection in the first frame that contains detections. No operator click is required. Useful for fixed-camera deployments where the target is known to be the dominant object in the scene.

Troubleshooting

SymptomCauseFix
nvbufsurface: Failed to create EGLImage nvinfer cannot find a usable EGL context. DISPLAY is either unset or a forwarded X11 socket. Ensure EGL_PLATFORM=surfaceless is set before any GStreamer import. The server sets this automatically — do not override it.
Client connects but video never appears (RTSP timeout after 20 s) Either GstRtspServer SDP generation failed (factory string wrong) or UDP RTP ports are blocked. Confirm rtph264pay name=pay0 is in the factory launch string. Confirm client uses protocols=tcp.
Click does not select correct object RTSP latency mismatch — click was resolved against a stale frame; or PGIE is disabled so no detections exist. Ensure PGIE is running (--no-pgie not set). Click latency compensation works best for slow-moving targets. Use select/next/lock workflow for precise selection.
[WAITING] No detections yet... persists PGIE model path is wrong or the engine was built for a different GPU. Check pgie_config.txt paths. Verify the engine exists: ls /opt/nvidia/deepstream/deepstream/samples/models/Primary_Detector/*.engine
Client GTK window opens but stays black GStreamer not yet connected or server not yet ready. Wait 5-10 s for RTSP negotiation and auto-reconnect. Check server is running and RTSP port 8554 is reachable: nc -zv <host> 8554
ModuleNotFoundError: pyds on server pyds Python bindings not installed. Install the pyds wheel for your DeepStream version from the JetPack SDK Manager or NVIDIA Developer Zone.
