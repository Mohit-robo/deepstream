# Phase 7: Hybrid Tracking Architecture & Testing Guide

This document outlines the design and verification procedures for the **SUTrack Hybrid Tracking System**, which integrates DeepStream's native multi-object tracking (`nvtracker`) with the high-precision SUTrack engine.

---

## 1. Architecture Overview

Phase 7 introduces a dual-tracker approach to solve the problem of ID instability and occlusion handling in high-precision scenarios.

### 1.1 Pipeline Topology
The GStreamer pipeline is structured as follows:
`Source` -> `PGIE (Detector)` -> `nvtracker (MOT)` -> `NvVideoConvert / Caps` -> `OSD (Probe)` -> `Sink`

- **PGIE (nvinfer)**: Generates initial detections (Car, Person, etc.) on frame 0 or continuously.
- **nvtracker**: Maintains identity for all objects in the scene using NvDCF (NVIDIA Data Association Tracker). It assigns "native IDs".
- **SUTrack (Probe-based)**: Injected into the OSD sink pad. It receives the RGBA frame, runs the high-precision SOT logic, and overlays stabilized metadata.

### 1.2 The Stabilization Logic
The "Hybrid" aspect comes from the correlation between the two trackers:
1. **Selection**: User clicks a box generated by PGIE/nvtracker.
2. **Benchmark**: SUTrack is initialized on that exact box and provides the "ground truth" precision.
3. **Correlation**: In every frame, the probe calculates the **IoU (Intersection over Union)** between SUTrack's high-precision box and all active native `nvtracker` objects.
4. **Correction**: If a match is found (IoU > 0.5), we associate the high-precision SUTrack box with that specific native ID. This ensures that even if native IDs fluctuate or objects cross, the system "locks on" to the physical object intended.

## 2. Shared Utilities (`app_utils.py`)

Boilerplate code has been migrated to a shared module to ensure consistency across Phase 6 and Phase 7 applications:
- **`select_bbox_click_to_select`**: Unified GUI for ROI selection from PGIE/Tracker detections.
- **`setup_rtsp_server`**: Shared factory for GstRtspServer initialization.
- **`get_iou`**: Math helper for cross-tracker matching.
- **`setup_logging`**: Standardized logging format.

## 3. Test Cases

### 3.1 TC1: Click-to-Select Initialization
- **Goal**: Verify that the system correctly initializes from an `nvtracker` ID.
- **Steps**:
    1. Run `python3 deepstream/apps/deepstream_nvtracker_app.py`.
    2. Wait for the first-frame GUI to appear.
    3. Click on a bounding box (e.g., a car).
- **Pass Criteria**: SUTrack initializes immediately, and the box border turns **Teal/Green** with the label `SUTrack [ID] (Stabilized)`.

### 3.2 TC2: ID Persistence (Occlusion Handling)
- **Goal**: Verify that the stabilization logic maintains a "lock" even if the background MOT fluctuates.
- **Steps**:
    1. Track an object that passes behind a tree or another object.
    2. Observe the native `nvtracker` ID (small white text) vs the SUTrack label (green box).
- **Pass Criteria**: Even if the native tracker misses a few frames or re-detects the object with a new ID, SUTrack should remain on the target. If the recovery occurs, the IoU logic should "re-match" the target to the correct identity.

### 3.3 TC3: Mid-stream Target Switching
- **Goal**: Verify the dynamic re-selection mechanism.
- **Steps**:
    1. Start tracking Object A.
    2. In the terminal, type `s` and press **ENTER**.
    3. The pipeline will pause visuals and show the selection GUI again.
    4. Click on Object B.
- **Pass Criteria**: SUTrack should drop Object A and begin tracking Object B without restarting the entire pipeline.

### 3.4 TC4: PGIE One-Shot Interaction
- **Goal**: Ensure detector optimization doesn't break the background tracker.
- **Steps**:
    1. Ensure `pgie_one_shot: true` is set in `tracker_config.yml`.
    2. Run the hybrid app.
- **Pass Criteria**: After the first selection, PGIE interval should be set to `10000` (verified in logs). `nvtracker` should continue to track the object based on visual cues and motion models (NvDCF) without needing continuous detection hits.
